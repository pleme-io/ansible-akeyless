# Auth once per play — `module_defaults` + `action_groups`

If you're using more than ~3 Akeyless modules in a play, repeating
`access_id` / `access_key` on every task gets noisy fast. The
collection ships `meta/runtime.yml` with an `action_groups.all:`
entry listing every module — so you can set the auth once per play
via `module_defaults`.

This was a frequent customer ask (the PRM-1767 ticket from the
Pepsico team): they didn't want every task to carry auth params
when the same credentials worked for every Akeyless call.

## The pattern

```yaml
- hosts: vault_clients
  module_defaults:
    group/drzln0.akeyless.all:
      gateway_url: "{{ akeyless_gateway_url }}"
      access_id: "{{ akeyless_access_id }}"
      access_key: "{{ akeyless_access_key }}"

  tasks:
    - drzln0.akeyless.role:
        name: app-readonly
        state: present

    - drzln0.akeyless.static_secret:
        name: /app/db/password
        value: "{{ rotated_password }}"
        state: present

    - drzln0.akeyless.target_db:
        name: /targets/prod-db
        db_type: postgres
        host: db.internal
        port: "5432"
        user_name: app
        pwd: "{{ db_admin_password }}"
        state: present
```

Each task above automatically receives `gateway_url`, `access_id`, and
`access_key` from the group default. Tasks that don't need those
fields just inherit them harmlessly.

## What `group/drzln0.akeyless.all:` does

Ansible's `module_defaults` accepts both single-module overrides
(`drzln0.akeyless.role:`) and group overrides
(`group/drzln0.akeyless.all:`). The `all` group is registered in
`meta/runtime.yml` (auto-generated, lists all 209 modules):

```yaml
action_groups:
  all:
    - account_settings_describe
    - account_settings_update
    - assoc_role_auth_method
    # ... 206 more ...
```

A `module_defaults` entry that targets the group applies to every
module in the list.

## Per-module overrides

You can still override per-task. Per-task wins:

```yaml
- hosts: vault_clients
  module_defaults:
    group/drzln0.akeyless.all:
      access_id: p-default

  tasks:
    # Uses access_id=p-default (group default)
    - drzln0.akeyless.role: { name: r1, state: present }

    # Uses access_id=p-override (per-task wins)
    - drzln0.akeyless.role:
        name: r2
        state: present
        access_id: p-override
```

## Combining with token caching

When you also wire the [cache plugin](token-caching.md), the auth
round-trip happens ONCE per playbook (cached for 25 minutes by
default), and `module_defaults` makes every task pick up the cached
token transparently.

## Verifying the group is current

The `action_groups.all` list is auto-generated by `ansible-forge`
during regeneration. If a new module ships and isn't in the group
list, `drzln0.akeyless.<new_module>` tasks won't inherit the
defaults.

A sanity test in this repo (`tests/sanity/test_action_groups_sync.py`)
pins this: every file in `plugins/modules/` must appear in
`meta/runtime.yml`'s `action_groups.all`. Drift fails CI.
